Prompt Protect

A drop-in HTTP proxy that sits between your backend and an LLM provider. Point your OpenAI-compatible client at it instead of OpenAI directly — no other changes required.

Your App  →  Prompt Protect  →  OpenAI

Every request is normalized, scanned for PII and secrets, scored for risk, and either allowed, masked, or blocked — before any data leaves your infrastructure.

Quick start

1. Configure

cp .env.example .env
# Add your OPENAI_API_KEY to .env

2. Run

docker compose up

Proxy on http://localhost:3000 · NER sidecar on http://localhost:5001

3. Verify

curl http://localhost:3000/health

Integration

Point any OpenAI-compatible client at the proxy by changing the base URL. The request format is identical to OpenAI's API.

curl http://localhost:3000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o","messages":[{"role":"user","content":"Hello!"}]}'
You'll need your OPENAI_API_KEY in the proxy's .env — the same key you already use in your app. The proxy injects it when forwarding to OpenAI. Your existing client code and env setup don't change.

Detection

Hybrid pipeline: regex detectors run first, then the NER sidecar for entity types that need linguistic context. Text is normalized before detection — Unicode tricks, zero-width characters, and Base64 blobs are collapsed first. The NER sidecar uses spaCy (en_core_web_sm by default; set SPACY_MODEL=en_core_web_trf for higher accuracy at the cost of ~2 GB RAM).

Detector Type Catches
EmailDetector :email user@example.com
PhoneDetector :phone 555-123-4567, +1 (800) 555 0100
AddressDetector :address 12 Main St, Springfield IL
IdDetector :id SSN 123-45-6789, credit card 4111 1111 1111 1111
IpDetector :ip 192.168.1.1, 2001:db8::1
SecretDetector :secret Bearer tokens, sk-... API keys, AWS access keys
DobDetector :dob DOB: 01/15/1990, born January 15 1990 (keyword required)
NerDetector :person Full names via NER sidecar
NerDetector :org Company and org names via NER sidecar
NerDetector :location Countries, cities, places via NER sidecar
If the NER sidecar is unavailable, detection falls back to heuristic regex-based person detection. Set NER_ENABLED=false to skip it entirely.

Risk levels

Rules are evaluated in priority order. First match wins.

Level Rule Condition
HIGH critical_type Any :id or :secret detected
HIGH identity_reconstruction :dob + :person together
HIGH mosaic_profile 3+ types from {person, org, location, email, phone, dob}
MEDIUM multiple_sensitive 2+ of {email, phone, address, ip}
MEDIUM single_sensitive 1 of {email, phone, address, ip, dob}
MEDIUM person_with_context :person + any other type
LOW person_only Name only, no other context
LOW clean No findings

Policy

Each risk level maps to a policy action. Configurable per level via env vars.

Risk Default Behaviour
LOW allow Request forwarded as-is
MEDIUM sanitize Sensitive values replaced with placeholders, then forwarded
HIGH block Request rejected with 422, LLM not called

Override defaults: PROMPT_PROTECT_POLICY_LOW · PROMPT_PROTECT_POLICY_MEDIUM · PROMPT_PROTECT_POLICY_HIGH

Valid values: allow · sanitize · block

Masking

When action is sanitize, sensitive values are replaced with stable typed placeholders.

James Carter         → [PERSON_1]
john@example.com     → [EMAIL_1]
555-123-4567         → [PHONE_1]
123-45-6789          → [ID_1]
192.168.1.1          → [IP_1]
sk-abc123...         → [SECRET_1]
DOB: 01/15/1990      → [DOB_1]

Placeholders are stable within a request — the same value always maps to the same placeholder, so the LLM response remains coherent. The placeholder-to-original mapping is returned in the response body.

Response headers

Every response includes these headers regardless of action taken.

Header Example Description
X-Prompt-Protect-Risk-Level medium Computed risk level
X-Prompt-Protect-Action sanitize Policy action taken
X-Prompt-Protect-Detected-Types email,person Comma-separated detected types
X-Prompt-Protect-Masked true Whether content was masked

Environment variables

Variable Required Default Description
OPENAI_API_KEY yes — Injected when forwarding to provider
OPENAI_API_BASE_URL no https://api.openai.com LLM provider base URL
PROMPT_PROTECT_POLICY_LOW no allow Action for low risk
PROMPT_PROTECT_POLICY_MEDIUM no sanitize Action for medium risk
PROMPT_PROTECT_POLICY_HIGH no block Action for high risk
NER_ENABLED no true Set false for regex-only mode
NER_SERVICE_URL no http://spacy:5001 NER sidecar URL
SPACY_MODEL no en_core_web_sm spaCy model — en_core_web_sm (fast) or en_core_web_trf (accurate, ~2 GB RAM)
CORS_ORIGINS no * Allowed CORS origins

Scenarios

Real request/response pairs for each major case.

Dry run

no API key needed

Add "dry_run": true to run the full pipeline without forwarding to OpenAI. Useful for testing and CI.

Request

{
  "model": "gpt-4o",
  "dry_run": true,
  "messages": [
    { "role": "user", "content": "My SSN is 123-45-6789" }
  ]
}

Response — 422 Blocked

{
  "error": "blocked",
  "risk_level": "high",
  "risk_explanation": {
    "rule": "critical_type",
    "reason": "Government or financial ID detected",
    "triggered_by": ["id"],
    "detected_values": ["123-45-6789"]
  },
  "messages": [
    {
      "role": "user",
      "findings": [{ "type": "id", "value": "123-45-6789" }]
    }
  ]
}

Sanitize — email address

MEDIUM

A single email triggers medium risk. The address is masked before the prompt reaches OpenAI.

Request

{
  "model": "gpt-4o",
  "messages": [
    { "role": "user", "content": "Forward the report to sarah.jones@acme.com" }
  ]
}

What gets forwarded to OpenAI

"Forward the report to [EMAIL_1]"

Response body (with LLM reply)

{
  "risk_level": "medium",
  "action": "sanitize",
  "messages": [
    {
      "role": "user",
      "masked_text": "Forward the report to [EMAIL_1]",
      "mapping": { "[EMAIL_1]": "sarah.jones@acme.com" },
      "findings": [{ "type": "email", "value": "sarah.jones@acme.com" }]
    }
  ],
  "choices": [ /* normal OpenAI response */ ]
}

Block — SSN

HIGH

Government IDs trigger critical_type immediately. Request is rejected, OpenAI is never called.

Request

{
  "model": "gpt-4o",
  "messages": [
    { "role": "user", "content": "Help me fill this form. SSN: 523-45-6789" }
  ]
}

Response — 422

{
  "error": "blocked",
  "risk_level": "high",
  "risk_explanation": {
    "rule": "critical_type",
    "reason": "Government or financial ID detected",
    "triggered_by": ["id"],
    "detected_values": ["523-45-6789"]
  }
}

Mosaic profile — name + org + location

HIGH

Three individually-low/medium fragments combine into a complete identity profile. Each field alone would be medium or low — together they cross the mosaic threshold.

Request

{
  "model": "gpt-4o",
  "messages": [{
    "role": "user",
    "content": "Write a LinkedIn bio for Sarah Chen, Senior Engineer at Stripe, San Francisco"
  }]
}

Response — 422

{
  "error": "blocked",
  "risk_level": "high",
  "risk_explanation": {
    "rule": "mosaic_profile",
    "reason": "3+ identity fragments form a complete profile",
    "triggered_by": ["person", "org", "location"],
    "threshold": 3
  }
}

API key in prompt

HIGH

Hardcoded credentials in pasted code are a common accidental leak. Detected as :secret and blocked.

Request

{
  "model": "gpt-4o",
  "messages": [{
    "role": "user",
    "content": "Why am I getting 401?\n\nconst client = new OpenAI({ apiKey: \"sk-proj-abc123xyz\" });"
  }]
}

Response — 422

{
  "error": "blocked",
  "risk_level": "high",
  "risk_explanation": {
    "rule": "critical_type",
    "reason": "Credential or secret detected",
    "triggered_by": ["secret"],
    "detected_values": ["sk-proj-abc123xyz"]
  }
}

Encoding evasion

MEDIUM

Fullwidth Unicode characters are used to disguise an email address. The text normalizer collapses them to ASCII before detection runs — bypasses all other tools, caught here.

Request — fullwidth Unicode email

{
  "model": "gpt-4o",
  "messages": [{
    "role": "user",
    "content": "Send the file to ｓａｒａｈ．ｊｏｎｅｓ＠ａｃｍｅ．ｃｏｍ"
  }]
}

After normalization (what detectors see)

"Send the file to sarah.jones@acme.com"

Response — sanitized

{
  "risk_level": "medium",
  "action": "sanitize",
  "messages": [{
    "masked_text": "Send the file to [EMAIL_1]",
    "mapping": { "[EMAIL_1]": "ｓａｒａｈ．ｊｏｎｅｓ＠ａｃｍｅ．ｃｏｍ" },
    "findings": [{ "type": "email", "value": "sarah.jones@acme.com" }]
  }]
}
